# 架构优化计划 — P1 (AI Gateway MVP 阶段)

Version: 1.0
Status: Draft
Owner: AI Project Manager
Last Updated: 2026-07-22

---

## 一、总结

当前项目在 P0 阶段设计了 5 个微服务，但实际只有 `gateway` 实现了业务逻辑，其余 4 个（`auth-service`、`billing-service`、`policy-engine`、`router-engine`）只是健康检查的 stub。同时 Redis 已部署但完全没有使用，导致每次请求都要穿透到数据库。

本计划的核心思路是：**在 P1 阶段先收（合并为单体），再补（加缓存），后稳（加容错）**。

---

## 二、当前状态分析

### 2.1 服务架构现状

```
backend/
  cmd/
    gateway/main.go          # ✅ 完整实现 (8080)
    auth-service/main.go     # ❌ stub (8083)
    billing-service/main.go  # ❌ stub (8084)
    policy-engine/main.go    # ❌ stub (8081)
    router-engine/main.go    # ❌ stub (8082)
```

4 个 stub 完全相同（只有 `/health` 端点），每个都引入全部 `internal/` 包但只用了 `HealthService`。

### 2.2 请求链路现状（以 chat 为例）

```
POST /v1/chat/completions
  │
  ├─ ValidateApiKey:    GetByPrefix(DB) + SHA256 verify        → 1 DB query
  ├─ CallWithFallback:
  │    ├─ modelRepo.GetByCode(DB)                               → 1 DB query
  │    ├─ bindingRepo.ListByModelID(DB)                         → 1 DB query
  │    └─ providerRepo.GetByID(DB) × N bindings                 → N DB queries
  │
  └─ RecordLog:         logRepo.Create(DB)                      → 1 DB query (同步)
```

**总计：4+N 次 DB 查询 + 同步写入日志**。与目标"99% 不查 DB、主链路 <10ms"差距较大。

### 2.3 Router Service 代码重复

`SelectProvider` 和 `CallWithFallback` 两个方法的前半段逻辑完全重复：
- GetByCode → ListByModelID → filter active → prefetch providers → sort

### 2.4 其他问题

| 问题 | 位置 | 影响 |
|------|------|------|
| HTTP Keep-Alive 关闭 | router_service.go:51 | 每次 provider 调用新建 TCP 连接 |
| 无熔断机制 | router_service.go | 失败 provider 会被反复调用 |
| 日志同步写入 | chat_controller.go:113-122, 146-157, 184-195 | 延长请求响应时间 |
| 无 Graceful Shutdown | gateway/main.go:159 | 重启中断请求 |
| 数据库双连接池 | postgres.go:25-51, 54-73 | 资源浪费 |

## 三、变更方案

### 3.1 Step 1: 合并 stub 服务为单一组件

**目标：** 消除 4 个无效 stub 服务的维护成本。

**涉及文件：**

| 操作 | 文件 | 说明 |
|------|------|------|
| 删除 | `backend/cmd/auth-service/main.go` | 无业务逻辑，仅 stub |
| 删除 | `backend/cmd/billing-service/main.go` | 同上 |
| 删除 | `backend/cmd/policy-engine/main.go` | 同上 |
| 删除 | `backend/cmd/router-engine/main.go` | 同上 |
| 删除 | `backend/Dockerfile` | 多阶段构建，`ARG SERVICE_NAME` 参数不再需要 |

**Docker 构建变更：**

旧的 Dockerfile 需要根据 `SERVICE_NAME` 构建不同二进制。合并后只需要一个固定入口，简化 Dockerfile：

```dockerfile
# backend/Dockerfile (简化后)
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY . .
RUN go mod download
RUN CGO_ENABLED=0 GOOS=linux go build -o /app/server ./cmd/gateway

FROM alpine:3.19
RUN apk --no-cache add ca-certificates
WORKDIR /app
COPY --from=builder /app/server .
COPY --from=builder /app/migrations ./migrations
EXPOSE 8080
CMD ["./server"]
```

**docker-compose.yml 变更：**

- 移除 `gateway` service 的 `depends_on: postgres, redis`（不需要等健康检查再启动）
- 其他 4 个 stub 对应的 docker-compose service 条目——**保留但注释掉**（等 P3+ 真正需要拆分时再启用），而不是直接删除，方便未来扩展
- `admin` service 的 `depends_on: gateway: condition: service_healthy` 保持不变

### 3.2 Step 2: Router 代码去重

**目标：** 消除 `SelectProvider` 和 `CallWithFallback` 中的重复逻辑。

**涉及文件：** `backend/internal/service/router_service.go`

**变更内容：**

提取一个私有方法 `resolveCandidates(ctx, modelCode)`，将以下逻辑抽出：

```
1. modelRepo.GetByCode → check status
2. bindingRepo.ListByModelID → filter active
3. prefetch providers → filter enabled
4. sort by priority + weight
```

然后 `SelectProvider` 和 `CallWithFallback` 都调用 `resolveCandidates`。

方法签名建议：
```go
type candidate struct {
    provider *entity.Provider
    binding  *entity.ModelProviderBinding
}

func (s *RouterService) resolveCandidates(ctx context.Context, modelCode string) (model *entity.Model, candidates []candidate, err error)
```

### 3.3 Step 3: 添加 Redis 缓存层

**目标：** 减少 DB 查询，使大部分请求走缓存。

**新的文件：** `backend/internal/repository/cache.go`

**实现方式：** 在 repository 层之上包装一个 cache-aside 模式：

```
读取:  cache.Get(key) → hit? → return
                        miss → DB → cache.Set(key, value, TTL)
写入:  DB → cache.Del(key) 或 cache.Set(key, value, TTL)
```

**初始缓存策略：**

| 缓存键 | 缓存内容 | TTL | 失效时机 |
|--------|---------|-----|----------|
| `apikey:{prefix}` | `{hash, user_id, status}` | 10min | 创建/更新/吊销 API Key 时删除 |
| `model:{code}` | `{id, name, status}` | 5min | 创建/更新/删除 Model 时删除 |
| `bindings:{model_id}` | `[{provider_id, weight, status}]` | 5min | 绑定/解绑时删除 |
| `provider:{id}` | `{name, base_url, api_key, priority, enabled}` | 5min | 创建/更新/删除 Provider 时删除 |

**涉及修改的文件：**

| 文件 | 修改内容 |
|------|---------|
| `internal/repository/cache.go` | **新建**：Redis 连接初始化 + 通用 Get/Set/Del 工具 |
| `internal/repository/api_key_repo_pg.go` | `GetByPrefix` 先查缓存，写入时同时更新缓存 |
| `internal/repository/model_repo_pg.go` | `GetByCode` 先查缓存；Create/Update/Delete 清除缓存 |
| `internal/repository/binding_repo_pg.go` | `ListByModelID` 先查缓存；Create/Delete 清除缓存 |
| `internal/repository/provider_repo_pg.go` | `GetByID` 先查缓存；Create/Update/Delete 清除缓存 |
| `internal/config/config.go` | 已有 `RedisURL` 和 `RedisPassword`，无需新增字段 |
| `internal/repository/postgres.go` | 添加 `NewCache()` 工厂函数 |

**注意：** 在 initial schema 中使用 `go-redis/redis` 或 `redis/go-redis` v9。

### 3.4 Step 4: 日志异步化

**目标：** 请求日志写入不再同步阻塞请求链路。

**方案：** 在 `UsageService` 中引入一个带缓冲的 channel 作为异步队列。

**涉及文件：**

| 文件 | 修改内容 |
|------|---------|
| `internal/service/usage_service.go` | 添加 `logChan chan *entity.RequestLog` + 后台 goroutine `startLogWorker()` |
| `internal/service/usage_service.go#RecordLog` | 改为向 channel 发送，不再同步 DB 写入 |

实现方式：
```go
const logChannelBuffer = 1000

func NewUsageService(...) *UsageService {
    svc := &UsageService{
        logChan: make(chan *entity.RequestLog, logChannelBuffer),
        ...
    }
    go svc.logWorker()
    return svc
}

func (s *UsageService) logWorker() {
    batch := make([]*entity.RequestLog, 0, 100)
    ticker := time.NewTicker(5 * time.Second)
    for {
        select {
        case log := <-s.logChan:
            batch = append(batch, log)
            if len(batch) >= 100 {
                s.flushLogs(batch)
                batch = batch[:0]
            }
        case <-ticker.C:
            if len(batch) > 0 {
                s.flushLogs(batch)
                batch = batch[:0]
            }
        }
    }
}
```

Channel 满时直接丢弃（或降级为同步写入），不阻塞主链路。

### 3.5 Step 5: HTTP 传输层优化 + 熔断（轻量级）

**目标：** Provider 调用更可靠。

**涉及文件：** `backend/internal/service/router_service.go`

**变更内容：**

1. **Keep-Alive 可配置化**：根据环境变量 `DISABLE_KEEP_ALIVE` 决定是否关闭，生产环境默认启用
2. **熔断机制**：在 `RouterService` 中添加一个 `sync.Map` 记录每个 provider 的连续失败次数，连续失败 >= 3 次时跳过该 provider 30 秒
3. **重试机制**：在 `CallWithFallback` 中，对可恢复错误（网络超时、5xx）等待 1s 后重试当前 provider 一次再 fallback

### 3.6 Step 6: Graceful Shutdown

**目标：** 重启服务时不中断进行中的请求。

**涉及文件：** `backend/cmd/gateway/main.go`

**变更内容：**

```go
srv := &http.Server{Addr: addr, Handler: handler}

go func() {
    if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        appLogger.Error("service stopped", slog.Any("error", err))
    }
}()

// Wait for interrupt signal
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
<-quit

appLogger.Info("shutting down...")
ctx, cancel := context.WithTimeout(context.Background(), 15*time.Second)
defer cancel()
srv.Shutdown(ctx)
```

## 四、前提与决策

### 4.1 依赖新增
- `github.com/redis/go-redis/v9` — Redis 客户端（Step 3 引入）

### 4.2 不做的决策
- **不做服务间 RPC**：合并为单体后，模块间直接函数调用即可，无需 gRPC/HTTP 调用
- **不做完整微服务拆分**：等到 P3+ 阶段流量增长到需要独立扩缩容时再拆
- **不做 Prometheus 指标**：放到 P2 阶段再做
- **不做 Rate Limiting**：放到 P2 阶段再做
- **不做完整 Event Bus**：先用 channel 实现异步日志，后续可升级到 Redis Stream

## 五、执行顺序与工作量

| 步骤 | 内容 | 工作量 | 前置依赖 |
|------|------|--------|---------|
| Step 1 | 合并 stub 服务 | ~1h | 无 |
| Step 2 | Router 去重 | ~1h | 无 |
| Step 3 | Redis 缓存 | ~4h | Step 1 |
| Step 4 | 日志异步化 | ~2h | Step 3（可选） |
| Step 5 | 传输层 + 熔断 | ~2h | 无 |
| Step 6 | Graceful Shutdown | ~0.5h | 无 |

**建议并行执行：**
- Step 1 + Step 2 + Step 6 可以同时做（互不依赖）
- Step 3 和 Step 4 依次做（Step 4 不依赖 Step 3，也可并行）
- Step 5 独立

**总计：** ~2 天 (以 P0 为标准: 1h = t-shirt size 估算)

## 六、验证方式

1. **Step 1**: 启动服务后访问 `GET /health` 和 `POST /api/v1/chat/completions` 均正常
2. **Step 2**: `SelectProvider` 和 `CallWithFallback` 行为不变，单元测试覆盖（可选）
3. **Step 3**: Redis 中有 `apikey:*` 和 `model:*` 等 key；重复请求不再产生 DB 查询
4. **Step 4**: 请求返回后日志延迟写入 DB（观察 `request_logs` 表）
5. **Step 5**: 停止一个 provider 服务，请求能自动 failover 到下一个 provider
6. **Step 6**: `kill -TERM <pid>` 后服务优雅关闭，进行中请求完成返回
