## 2. Context

Nova AI Gateway 需要在 P0 到 P2 阶段快速完成从基础设施到 AI Gateway MVP、再到 Policy Platform 的持续迭代。当前技术选型不仅影响工程实现，也直接影响部署复杂度、Agent 协作成本和后续扩展能力。

### 业务背景

- 一人公司需要优先选择成熟、稳定、低运维复杂度的技术
- 产品目标是面向开发者和企业提供统一 AI Gateway 能力
- 需要兼顾快速上线、成本优化和未来扩展

### 技术背景

- 系统包含后端 API 服务、管理后台、数据库、缓存和基础设施编排
- 项目已有 `11-coding-standard.md` 明确建议技术栈和仓库结构
- 需要保证与 AI Company 的标准、模板和文档体系一致

---

## 3. Problem

需要为 Nova AI Gateway 确定 P0 到 P2 阶段的核心技术栈，使系统能够快速启动、稳定交付、便于维护，并为未来扩展预留空间。

### 问题描述

如果技术栈选择过重，会增加一人团队的负担；如果过轻或不成熟，会影响稳定性、性能目标和后续扩展。

### 问题影响

| 影响方面 | 说明 |
|---------|------|
| 功能 | 直接决定开发速度和交付能力 |
| 性能 | 决定 Gateway、Policy、Router 的延迟和吞吐能力 |
| 成本 | 决定基础设施、运维和人力成本 |
| 开发效率 | 决定 Agent 输出代码和文档的一致性 |

---

## 4. Constraints

| # | 约束 | 类型 | 说明 |
|---|------|------|------|
| 1 | Gateway 主链路必须 < 10ms（不含模型推理） | 技术 | 高性能是硬指标 |
| 2 | Policy / Router 决策必须 < 2ms | 技术 | 决策链路必须足够轻 |
| 3 | 第一阶段部署必须基于 Docker Compose | 技术 | 明确不使用 K8S |
| 4 | 技术栈必须适合一人团队维护 | 资源 | 学习和运维成本必须可控 |
| 5 | 必须与现有 Coding Standard 保持一致 | 技术 | 避免标准与实现脱节 |

### 不可妥协的约束

- 不引入当前阶段不必要的复杂中间件
- 不选择与现有标准冲突的技术栈

### 可协商的约束

- 监控、消息队列、对象存储等可按阶段补充
- 向量数据库、Kubernetes 等延后到后续阶段评估

---

## 5. Options

| # | 方案名称 | 描述 | 优点 | 缺点 | 可行性 |
|---|---------|------|------|------|:------:|
| A | Go + Vue3 + PostgreSQL + Redis + Docker Compose | 使用标准中已定义的轻量高性能组合 | 性能好、生态成熟、与标准一致、适合一人团队 | 需要自己建设较多基础能力 | 高 |
| B | Node.js 全栈 + MongoDB + Redis | JS 技术栈统一 | 开发快、语言统一 | 对高性能网关链路不如 Go 稳定，文档标准不一致 | 中 |
| C | Java / Spring Boot + React + MySQL + Redis + Docker Compose | 企业常见组合 | 企业生态成熟 | 过重，启动慢，开发和运维成本高 | 低 |

---

## 6. Decision

选择 **方案 A：Go + Vue3 + PostgreSQL + Redis + Docker Compose**。

**决策**：
- 后端：Go 1.22+
- 前端：Vue3 + TypeScript + Pinia
- 数据库：PostgreSQL 15+
- 缓存：Redis 7+
- 部署：Docker Compose
- 包管理：Go Modules + pnpm

**决策日期**：2026-07-20

**决策人**：Architect

---

## 7. Why

这一技术栈的核心优势在于：性能足够、复杂度可控、生态成熟、与现有标准完全一致。

### 决定性因素

| 因素 | 本方案 | 其他方案 | 说明 |
|------|--------|---------|------|
| Gateway 性能 | 强 | 中/弱 | Go 更适合高吞吐、低延迟 API 网关 |
| 开发维护成本 | 低 | 中/高 | Vue3 + Go 的组合简单直接，适合一人团队 |
| 与标准一致性 | 完全一致 | 部分或不一致 | 现有 Coding Standard 已明确该组合 |
| 部署复杂度 | 低 | 中/高 | Docker Compose 足以覆盖 P0-P2 |

### 与约束的匹配

| 约束 | 满足情况 | 说明 |
|------|---------|------|
| Gateway 性能要求 | ✅ | Go 能较好满足低延迟目标 |
| Policy / Router 轻量决策 | ✅ | Go + Redis 缓存适合快速决策 |
| Docker Compose 部署 | ✅ | 直接满足第一阶段部署要求 |
| 一人团队维护 | ✅ | 技术栈简单且社区成熟 |
| 与标准一致 | ✅ | 完全按 `11-coding-standard.md` 执行 |

---

## 8. Tradeoffs

### 接受的代价

| 代价 | 影响 | 接受理由 |
|------|------|---------|
| 前后端不是同一语言 | 需要维护 Go 和 TypeScript 两套技术栈 | 性能和工程边界收益更高 |
| 不引入重型平台能力 | 需要后续自行补监控、消息、对象存储能力 | 当前阶段应避免过度设计 |

### 放弃的优势

- 放弃了 Node.js 全栈的单语言便利性
- 放弃了 Spring Boot 在企业生态中的现成中台能力

---

## 9. Consequences

### 正面后果

- 后端链路更容易满足性能目标
- 管理后台和工程规范更清晰
- Docker Compose 能快速启动开发、测试和生产初版环境
- 文档、代码和 Agent 输出能直接与标准对齐

### 负面后果

- 需要维护多语言开发环境
- 某些高级基础设施能力需要后续阶段再补足

### 迁移方案

当前无需迁移。后续演进原则：
- P3 之前不引入 Kubernetes
- P4/P5 之后根据规模评估对象存储、消息队列、向量数据库
- P6 再评估 Kubernetes 或多节点高可用

## 10. Risks

| # | 风险描述 | 等级 | 可能性 | 影响 | 缓解方案 |
|---|---------|------|--------|------|---------|
| 1 | Go 后端工程骨架搭建初期成本高于脚手架型框架 | 中 | 中 | 中 | 通过清晰目录结构和模板降低启动成本 |
| 2 | 前后端并行时接口契约不明确 | 中 | 中 | 高 | 提前建立 API 设计规范和 Architecture 文档 |
| 3 | Docker Compose 环境与后期生产环境出现偏差 | 中 | 中 | 中 | 将 Compose 作为第一阶段权威基础设施定义 |

| 方案 | 未选原因 | 未来可行性 |
|------|---------|:----------:|

| Java / Spring Boot | 对当前阶段过重，运维和开发成本高 | 中 |
